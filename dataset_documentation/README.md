# Dataset Documentation

This folder documents the datasets generated from the custom 5G-LENA digital twin scripts in
`contrib/nr/examples/5G-LENA-digital-twin-script/`.

The documentation is organized around two base simulation scenarios:

- **Bursty traffic scenario**: generated by `cellular-network-user.cc`
- **Delay benchmarking scenario**: generated by `delay-benchmarking-user.cc`

Here, **scenario** means the base simulation setup and traffic mix. The individual `runXX` and
`benchmarkXX` entries are experiment runs created by changing selected parameters within those
base scenarios.

## Common 5G Network Setup

Both scenarios use the same shared 5G network model through the `CellularNetwork(...)` setup:

- one gNB and a configurable number of UEs
- single component carrier and single bandwidth part
- 3.5 GHz carrier frequency and 40 MHz bandwidth
- numerology 1
- TDD pattern `DL|DL|DL|S|UL` repeated across the 10 ms frame
- OFDMA proportional-fair scheduler
- realistic beamforming triggered from SRS measurements
- UE uplink power control enabled
- ideal RRC enabled
- gNB antenna array of 4 rows x 8 columns and UE antenna array of 2 rows x 4 columns

## Scenario 1: Bursty Traffic

This scenario is generated by `cellular-network-user.cc`. It is intended to create application-level
bursty uplink traffic while also collecting RAN, queueing, and application delay traces.

The campaign uses:

- 10 UEs by default
- synthetic bursty traffic enabled by default
- selected UEs running the bursty traffic app in uplink
- UL and DL delay probe applications
- application generation time of 1000 seconds in the campaign runner

### Bursty Traffic Experiment Runs

All runs inherit the common campaign defaults:

| Run | Parameter change from campaign default | Purpose |
|---|---|---|
| `run01` | none | Baseline bursty run |
| `run02` | `numUesWithVrApp=2` | Lower bursty traffic load |
| `run03` | `numUesWithVrApp=5` | Higher bursty traffic load |
| `run04` | `numUesWithVrApp=6` | Highest bursty traffic load in this campaign |
| `run05` | `channelScenario=InH-OfficeMixed` | Indoor channel variation |
| `run06` | `channelScenario=UMi` | Urban micro channel variation |
| `run07` | `channelScenario=UMa` | Urban macro channel variation |
| `run08` | `vrAppProfile=Minecraft` | Bursty traffic profile variation |
| `run09` | `vrAppProfile=GoogleEarthVrCities` | Bursty traffic profile variation |
| `run10` | `vrAppProfile=GoogleEarthVrTour` | Bursty traffic profile variation |

## Scenario 2: Delay Benchmarking

This scenario is generated by `delay-benchmarking-user.cc`. It is intended to study uplink delay
components by sending controlled UDP delay probes, with optional background load.

The scenario differs from the bursty traffic scenario in these main ways:

- it uses fixed UE positions instead of mobile UEs
- no bursty traffic app is installed
- the number of UEs is set from the load mode: 1 UE with no background load, 2 UEs when TCP or UDP
  background load is enabled

### Delay Benchmarking Experiment Runs

All runs inherit the common benchmark defaults:

| Run range | Parameter sweep | Background load | Purpose |
|---|---|---|---|
| `benchmark01`-`benchmark09` | `delayPacketSize` in `{30, 100, 1200}` bytes and `delayInterval` in `{20ms, 50ms, 100ms}` | none | Baseline delay behavior without competing traffic |
| `benchmark10`-`benchmark18` | same packet-size and interval sweep | TCP background load | Delay behavior under TCP uplink load |
| `benchmark19`-`benchmark27` | same packet-size and interval sweep | UDP CBR load, `cbrLoad=10` Mbps | Delay behavior under moderate UDP uplink load |
| `benchmark28`-`benchmark36` | same packet-size and interval sweep | UDP CBR load, `cbrLoad=20` Mbps | Delay behavior under higher UDP uplink load |

## Data Formats

Each generated run can be processed into multiple data formats:

| Format | Description | Documentation |
|---|---|---|
| Raw ns-3 logs | Direct trace files emitted by the simulator | [raw_ns3_data_documentation.md](raw_ns3_data_documentation.md) |
| Parsed uplink data | Windowed per-RNTI and cell-level features generated from raw logs | [parsed_uplink_data_documentation.md](parsed_uplink_data_documentation.md) |
| Delay decomposition data | Per-packet uplink delay component table generated from joined trace files | [delay_decomposition_data_documentation.md](delay_decomposition_data_documentation.md) |

For each generated run, the campaign runner writes the exact `ns3 run` command into
`sim_campaign_logs/<run>/run_cmd.txt`.
